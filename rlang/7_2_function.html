Function fundamentals

幾個重要觀念

  • 函數也是物件,就像是vectors 也是物件。
  • 由三個部份組成: arguments, body, and environment.

但是規則都會被打破,例外大概都利用C寫成的 "primitive" base functions。

First-class functions

在R中,函數也是物件,這種特性也叫做 "first-class functions".

例如在list中,也可以放入函數:

funs <- list(
  half = function(x) x / 2,
  double = function(x) x * 2
)

funs$double(10)
#> [1] 20

匿名函數:

lapply(mtcars, function(x) length(unique(x)))
Filter(function(x) !is.numeric(x), mtcars)
integrate(function(x) sin(x) ^ 2, 0, pi)
調用匿名函數

如果要調用匿名函數,參考下面的方式

(function(x) 3)()

在R語言中,函數有時叫做closure因為,因為R函數包含(enclose)它們的環境 environments.

01<-function() {3}
typeof(f01)
#> [1] "closure" #🏷👏對照後面class的結果
str(f01)
#> function ()  
#>  - attr(*, "srcref")= 'srcref' int [1:8] 1 6 1 19 6 19 1 1
#>   ..- attr(*, "srcfile")=Classes 'srcfilecopy', 'srcfile' <environment: 0x00000000188ac3c8="">
class(f01)
#> [1] "function" #🏷👏是function,同時繼承自closure
#> [1] "closure"

函數的組成

1個函數有3個部分:

  • formals(), 參數
  • body(), {}內部.
  • environment(), 決定函數怎樣找出變數(names)的內容。.

測試3部分

f02 <- function(x) {
  3
}

formals(f02)
#> $x

body(f02)
#> {
#>     3
#> }

environment(f02) #🏷👏本函數f(02)會去搜尋的變數空間
#> <environment: 0x0000000019cdc128>

👍就像所以其他R的物件,函數也有很多 attributes(). 其中一個 "srcref", 是 source reference的縮寫。 　

attr(f02, "srcref")
#> function(x) {
#>   3
#> }
例外

但是有些函數像是 sum() and [，這些所謂的原始函數(Primitive functions)，無法利用上面的三個函數顯示其內容,因為這些函數存在於基本套件(base packages) 而且通常用Ｃ語言寫成。

sum
#> function (..., na.rm = FALSE)  .Primitive("sum")
`[` # <!--html_preserve--><span  class="codenote nopre">🏷<span   class="tooltiptext" >👏這也是函數</span></span><!--/html_preserve-->
#> .Primitive("[")

看一下type 分別屬於 "builtin" or "special":

typeof(sum)
#> [1] "builtin"
typeof(`[`)
#> [1] "special"

因為build-in， 所以這些函數的formals(), body(), and environment() 都回傳 NULL:

formals(sum)
#> NULL
body(sum)
#> NULL
environment(sum)
#> NULL
attr(sum, "srcref") # <!--html_preserve--><span  class="codenote nopre">🏷<span   class="tooltiptext" >👏C寫的,自然也沒有參考程式碼</span></span><!--/html_preserve-->
#> NULL

變數和函數的搜尋

R的's lexical scoping 遵循4個主要規則::

  • Name masking
  • Functions vs. variables
  • A fresh start
  • Dynamic lookup

下面的執行結果傳回10還是20?

x <- 10
g01 <- function() {
  x <- 20
  x
}

g01()
20

Name masking

內部範圍的宣告(第一次使用)覆蓋外部範圍的宣告。.

x <- 10
y <- 20
g02 <- function() {
  x <- 1
  y <- 2
  c(x, y)
}
g02()
#> [1] 1 2

如果在內部宣告找不到,就找外一層，一直到global environment。

x <- 2
g03 <- function() {
  y <- 1
  c(x, y)
}
g03()
#> [1] 2 1

上面的規則仍然適用於函數中的函數. 例如下例中的函數i()

x <- 1
g04 <- function() {
  y <- 2
  i <- function() {
    z <- 3
    c(x, y, z)
  }
  i()
}
g04() # 🏷👏g04() returns c(1, 2, 3).

同樣也適用於建立函數的函數( closures); 下例中的g06()

x <- 10
y <- 20

g05 <- function() {
  y <- 2
  function() {
    c(x, y)
  }
}
g06 <- g05()
g06() # 🏷👏g06() returns c(10, 2).

既然函數也只是普通的物件,那麼同樣的名稱尋找規則也適用於函數。

例如下面這個例子中,g07在外部和內部皆有定義。

g07 <- function(x) x + 1 # 🏷👏全局g07()
g08 <- function() {
  g07 <- function(x) x + 100 # 🏷👏局部g07()
  g07(10) #🏷👏這裡調用的是局部g07()
}
g08()
#> [1] 110

但是如果同一個名稱,在不同範圍有不一樣的型態呢?例如g9一個是變數,一個是函數:

g09 <- function(x) x + 100
g10 <- function() {
  g09 <- 10
  g09(g09)
}
g10()
#> [1] 110

一般來講,上面的用法在語法上是沒問題,但是最好避免。

A fresh start

問題:下例中的函數g11()第一次執行和第二次執行有甚麼不同?


g11 <- function() {
  if (!exists("a")) {
    a <- 1
  } else {
    a <- a + 1
  }
  a
}

g11()
g11()

❓g11() 每次被調用都是傳回 1。為甚麼?
每次執行g11()的時候,一個新的environment會被建立,也就是這個函數的局部空間。 而這個局部空間會在函數結束的時候✔️消失。

函數 exists(x) :會尋找變數名稱x是否存在,存在則無回 TRUE,否則傳回 FALSE. 而exists(x)的搜尋路徑,可以用search()✔️知道。

Dynamic lookup

但是如果函數執行的時候在局部環境找不到的變數的時候,會於目前的總體環境(也就是函數所在的環境)找變數值。

g12 <- function() x + 1 #🏷👏g12()定義在總體環境中
x <- 15 #🏷👏x定義在總體環境中
g12()
#> [1] 16

x <- 20
g12()
#> [1] 21
展開

上面的函數g12()用到變數x，因此需要在搜尋路徑中,有變數x的存在。因此在第二行中,定義了x,這個x存在於外部總體環境中。

類似這種情況,如果是故意的,那就不需檢查。但如果不是特意的,有可能會出錯。這種變數稱為unbound symbols(簡單講,就是是不宣告而使用)。 如果要檢查函數中有沒有這種變數，可以使用工具 codetools::findGlobals(). 這個函數可以列出函數中是否有unbound symbols :

codetools::findGlobals(g12)
#> [1] "+" "x"

也可以不依賴套件,而利用函數 emptyenv() :

g12 <- function() x + 1 #<!--html_preserve--><span  class="codenote nopre">🏷<span   class="tooltiptext" >👏此時,environment(g12) 為 &lt;environment: R_GlobalEnv&gt;</span></span><!--/html_preserve-->
x=4
environment(g12) <- emptyenv() # <!--html_preserve--><span  class="codenote nopre">🏷<span   class="tooltiptext" >👏此時,environment(g12) 為 &lt;environment: R_EmptyEnv&gt;</span></span><!--/html_preserve-->
g12()
#> Error in x + 1: 沒有這個函式 "+"

👍environment(g12) <- emptyenv()這一行的意思👺不是清掉g12()所在環境的的變數,而是將emptyenv()這個新空間指派給g12()讓他不要去搜尋總體環境(R_globalEnv)。

參數

Lazy evaluation

lazily evaluated: 如果參數沒有在函數中被存取,就不會執行。

  • 利用debug觀察 
        h01 <- function(x) {
      10
    }
    h01(x=runif(10))
  • 另一個例子,這裡的stop()不會被執行。因為沒用到參數x 
    h01 <- function(x) {
  10
}
h01(stop("This is an error!"))

Forcing evaluation

強迫參數計算並取值可以利用函數 force():

h02 <- function(x) {
  force(x)
  10
}
h02(stop("This is an error!"))

  1. 看一下為甚麼需要force()

    capture1 <- function(x) {
  function() {
    x
  }
}

  2. 由於lazy evaluation ,在調用capture1()的時候,x不會被執行

    x <- 10
h03 <- capture1(x) #🏷👏 x沒有被執行
h04 <- capture1(x) #🏷👏 x沒有被執行

h03() #🏷👏 x在這裡evaluated
#> [1] 10
x <- 20
h04() #🏷👏 x在這裡evaluated
#> [1] 20

  3. 但是當我們調用h03(),h04()第二次的時候,都是第一次放進去的值

    x <- 30
h03()
#> [1] 10 #🏷👏 x仍然是第一次的值
h04()
#> [1] 20

  4. 上面是因為lazy evaluation. 在第一次執行以後,就被cached。因此我們需要 forcing x:

    capture2 <- function(x) {
  force(x)

  function() {
    x
  }
}

x <- 10
h05 <- capture2(x)

x <- 20
h05()
#> [1] 10

預設參數值

h07x <-function(x=1){
  x+1
}
h07x() # return 2
#> [1] 2

h07 <- function(x = 1, y = x * 2) { # 🏷👏第二個參數,利用了第一個參數
  c(x, y)
}

h07()
#> [1] 1 2

缺失參數值Missing arguments

missing(): 用來判斷參數內容來自使用者或者預設的函數

h09 <- function(x = 10) {
  list(missing(x), x)
}
str(h09()) # 預設
#> List of 2
#>  $ : logi TRUE #🏷👏因為使用者沒傳參數,用了預設值,因者是TRUE
#>  $ : num 10
str(h09(10)) # 使用者
#> List of 2
#>  $ : logi FALSE
#>  $ : num 10

脫離函數

函數結束的時候,大部分是下面兩個狀態中的一種:

  • 傳回值: 表示成功執行,這裡討論
    • implicit vs explicit.
    • visible vs. invisible.
  • 丟出錯誤訊號: 表示無法執行。討論
    • 錯誤發生時的處理。函數的錯誤脫離狀態可以是產生一個旗號,用來觸發既有的處理函數,可以是重新啟動,或是互動訊息向是按Q脫離。

Implicit vs. explict returns

函數的返回值有兩種方式:

  • Implicitly, 最後一個 expression 變成傳回值:

    f <- function(x) {
  if (x < 10) {
    0
  } else {
    10
  }
}
f(5)
#> [1] 0
f(15)
#> [1] 10

  • Explicitly, 利用 return():

    f <- function(x) {
  if (x < 10) {
    return(0)
  } else {
    return(10)
  }
}

Invisible values

大部分函數再傳回值的時候都會顯示在主控台上:

f <- function() 1
f()
#> [1] 1

但是也可以不用顯示:利用函數 invisible()。

j04 <- function() invisible(1)
j04()

這種不會顯示傳回值的函數,可以用下面的方法驗證:

print(j04())
#> [1] 1
(j04())
#> [1] 1
# 或者利用函數 `withVisible():
str(withVisible(j04()))
#> List of 2
#>  $ value  : num 1
#>  $ visible: logi FALSE

✔️在R語言中,傳回值是不可見的函數是 <-:

a <- 2
(a <- 2)
#> [1] 2
a <- b <- c <- d <- 2

一般而言，像是 ( <-, print(), or plot()) 都是傳回不可見結果。

Errors

函數可以利用函數 stop()結束一個函數的執行。

j05 <- function() {
  stop("I'm an error")
  return(10)
}
j05()
#> Error in j05(): I'm an error

Exit handlers

設定函數執行完畢時要執行的結束函數。例如,有些設定值,在本函數中被暫時修改,但是在結束執行的時候,想要回復原狀，或者要清除本函數產生的一些垃圾檔案,都可以設定這脫離函數.這種脫離函數,不管函數執行完畢時,是否錯誤,都會執行。

f <- function(x) {
  cat("Hello\n")
  on.exit(cat("Goodbye!\n"), add = TRUE)
  
  if (x) {
    return(10)
  } else {
    stop("Error") #🏷👏產生錯誤訊號,會結束函數執行
  }
}

f(TRUE)
#> Hello
#> Goodbye!
#> [1] 10

f(FALSE)
#> Hello
#> Error in f(FALSE): Error
#> Goodbye!
on.exit() 的第二個參數幾乎都是設定為add = TRUE。如果不設定為T,先前的 exiting handler 會被蓋過去。

on.exit() 清除中間檔案的範例:

cleanup <- function(dir, code) {
  old_dir <- setwd(dir)
  on.exit(setwd(old), add = TRUE)
  
  old_opt <- options(stringsAsFactors = FALSE)
  on.exit(options(old_opt), add = TRUE)
}

範例:某個函數必須切換工作目錄,然後執行某些工作,但是在函數執行完畢後,要設定為原先的工作目錄,則

with_dir <- function(dir, code) {
  old <- setwd(dir) # 🏷👏會把目前的工作目錄記錄在old
  on.exit(setwd(old), add = TRUE) # 🏷👏切換回去

  force(code)
  
}

getwd()
#> [1] "D:/work/rmi_r/content/rlang"
with_dir("~", getwd())
#> [1] "C:/Users/linchao/Documents"

指定多個on.exit()函數的時候,會按照指定的順序FIFO執行。

f <- function() {
  on.exit(message("a"), add = TRUE)
  on.exit(message("b"), add = TRUE)
}
f()
#> a
#> b

在FILO的演算法中,前述的FIFO會造成困擾,可以利用 after = FALSE,來放在最前面:

f <- function() {
  on.exit(message("a"), add = TRUE, after = FALSE)
  on.exit(message("b"), add = TRUE, after = FALSE)
}
f()
#> b
#> a
